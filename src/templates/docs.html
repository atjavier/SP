Getting Started

Input Requirements

  • Accepted files: VCF and VCF.GZ inputs containing human single-nucleotide variants (SNVs). Multi-allelic sites are split into per-allele records when applicable. Non-SNV entries are flagged or excluded.
  • Core fields: Standard VCF headers (#CHROM, POS, REF, ALT) are required. Sample columns are accepted if present.
  • Reference build: The run stores a reference build value that should match your inputs and downstream evidence sources. Mismatched builds can cause missing evidence or incorrect IDs.
  • Normalization: Provide left-aligned, normalized SNVs whenever possible so evidence lookups align with database records.

Pre-run Checklist

  • VCF validation: Fix malformed headers or invalid REF/ALT fields before upload.
  • Evidence mode: Choose online, offline, or hybrid based on available local databases.
  • Failure policy: Decide whether evidence failures should stop the run or continue with partial data.

Run Basics

  • Run flow: Upload -> validate -> pipeline stages execute sequentially with live status updates.
  • Run ID: Each run receives a unique identifier used for logs, stages, and results retrieval.
  • Scope: {{ app_name }} is built for research and teaching workflows, emphasizing transparency over clinical diagnosis.

Pipeline Stages

Parser

  • What it does: Validates the VCF, extracts SNV records, and rejects unsupported entries.
  • Key outputs: Normalized SNV table, exclusion list for non-SNV or malformed variants.
  • Where to view: Parser tab summarizes counts and exclusions for the run.
  • Why it matters: Ensures downstream tools only receive analyzable SNVs.

Pre-Annotation

  • What it does: Computes base-change context (REF>ALT), substitution class, and nucleotide classes.
  • Key outputs: Base change, transition/transversion labels, ref/alt class fields.
  • Where to view: Pre-annotation tab lists per-variant base changes and classes.
  • Why it matters: Feeds classification and determines eligibility for prediction tools.

Classification

  • What it does: Uses VEP consequence terms to assign variant class (missense, synonymous, nonsense, other).
  • Key outputs: Classification category plus a reason code that explains the decision.
  • Where to view: Consequence tab and the variant details panel show category and reason.
  • Why it matters: Determines which variants proceed to functional prediction.

Prediction

  • What it does: Runs SIFT, PolyPhen-2, and AlphaMissense predictors through VEP plugins where configured.
  • Key outputs: Predictor scores, labels, source metadata, and reason messages when unavailable.
  • Where to view: Functional prediction tab and the variant details panel list per-tool outputs.
  • Why it matters: Provides functional impact estimates for missense variants.

Annotation

  • What it does: Pulls evidence from dbSNP, ClinVar, and gnomAD (online and/or local VCF sources).
  • Key outputs: Evidence records with outcome, completeness, source, and retrieval timestamps.
  • Where to view: Evidence annotation tab shows dbSNP, ClinVar, gnomAD tables and annotated VCF preview.
  • Why it matters: Anchors variant interpretation with known identifiers and population context.

Reporting

  • What it does: Aggregates stage outputs into run-level summaries and evidence diagnostics.
  • Key outputs: Highlights, significant results, and completeness summaries.
  • Where to view: Reports tab contains summaries, evidence diagnostics, and HTML artifacts when available.
  • Why it matters: Provides a final, consolidated view of the run for review and export.

Tools and Evidence Sources

Consequence Annotation

  • VEP: Required for classification and prediction. Produces transcript-level consequence terms, gene context, and plugin outputs that feed downstream stages.
  • SnpEff: Optional local annotation step for additional effect context and diagnostics; results are tracked in the annotation stage when enabled.
  • Configuration note: VEP relies on a local cache and optional FASTA/plug-in paths. Missing or invalid configuration causes Classification and Prediction to fail with a clear error message.
  • Local databases: SnpEff uses a local genome database; missing databases are reported in stage results.

Functional Prediction

  • SIFT: Scores amino acid substitutions and labels them as tolerated or deleterious.
  • PolyPhen-2: Provides probability-based impact predictions with categorical labels.
  • AlphaMissense: Deep learning-based pathogenicity scoring via VEP plugin data when configured.
  • Applicability: Prediction outputs are most relevant for missense variants; other classes are flagged as not evaluated.
  • Integration: Predictor outputs arrive through VEP plugin fields and are surfaced with per-tool reason messages.

Evidence Databases

  • dbSNP: Provides rsIDs and metadata for variant identity and cross-references.
  • ClinVar: Supplies clinical significance assertions and stable ClinVar identifiers when available.
  • gnomAD: Adds population allele frequency context for prioritization and filtering.
  • Source modes: Each database can be queried online or via local VCFs; outcomes distinguish not found (no record) from unavailable (source unreachable or misconfigured).

Evidence Modes and Policy

Modes

  • Online: Query remote APIs for dbSNP, ClinVar, and gnomAD evidence.
  • Offline: Use locally configured, tabix-indexed VCF databases for evidence lookups.
  • Hybrid: Combine local sources with online fallback where configured.
Evidence mode is chosen before starting a run (or before retrying Annotation) and applies to all variants.

Failure Policy

  • Stop: Fail the annotation stage and block reporting until evidence issues are resolved.
  • Continue: Allow the run to proceed with evidence marked as partial or unavailable.
  • Transparency: Every failure is surfaced with a reason message and troubleshooting hint.
Evidence completeness is tracked per source and summarized in Reporting and the Results tables.

Results Interpretation and Provenance

Progress vs Results

  • Progress tab: Run summary, stage timeline, logs, and retry/cancel actions.
  • Results tab: Stage outputs, evidence diagnostics, and reporting summaries.

How to Read Results

  • Reports Summary: Consolidated narrative of the run outcomes and major findings.
  • Variant Summary: Per-variant table listing classification, prediction, evidence, and completeness.
  • Stage tabs: Each stage exposes its own outputs so you can trace how results were produced.
  • Variant details panel: Click a variant row to see per-tool evidence, scores, and reasons.
  • Evidence diagnostics: Reports tab shows source-level completeness and any evidence warnings.
  • HTML artifacts: When available, the Reports tab embeds rendered summaries (tables or plots).

Completeness and Provenance

  • Completeness: Complete means all configured sources responded, partial means one or more sources are missing, unavailable means no sources responded, and failed indicates a stage-level error.
  • Outcome fields: Each tool/source reports success, not found, unavailable, or error outcomes.
  • Reason fields: Human-readable messages explain missing data, timeouts, or configuration gaps.
  • Provenance fields: Each record includes tool/source labels and retrieval timestamps.
All tables and controls are keyboard-navigable with visible focus styles.

Troubleshooting and FAQ

  • Validation fails immediately: Confirm the VCF header and REF/ALT fields are valid SNVs.
  • No runs visible: Upload a VCF and press Start. The run summary populates after validation.
  • Classification fails: Verify VEP configuration and cache paths, then retry the Classification stage.
  • Prediction missing outputs: Check VEP plugin setup (SIFT, PolyPhen-2, AlphaMissense) and rerun Prediction.
  • Evidence sources unavailable: Check connectivity or local VCF paths; retry Annotation once resolved.
  • gnomAD lookup errors: Ensure API connectivity or local dataset compatibility for the configured build.
  • Partial results: Use Evidence diagnostics to identify which sources failed and why.
  • Runs look stale after reload: Reopen the last run in the Results tab to refresh summaries.
  • Reference build mismatch: Re-run the pipeline with the correct build to improve evidence matches.
