How to Use the App

A practical, task-focused walkthrough of a typical run. Follow this first if you are new.

Quick Start (5 Steps)

  1. Select a VCF or VCF.GZ file in the Start card.
  2. Pick an evidence failure behavior (stop on failure or continue with partial evidence).
  3. Press Start to validate and launch the pipeline.
  4. Watch the Progress tab for stage status, timings, and logs.
  5. Open the Results tab to inspect stage outputs and summaries.

During a Run

  • Retry failed stage: Appears when a stage fails and the run is not active.
  • Cancel run: Stops an active run safely and records the canceled state.
  • Run logs: Show the latest log lines and auto-refresh while running.

After a Run

  • Stage tabs: Parser, Pre-annotation, Consequence, Prediction, and Evidence annotation show raw outputs.
  • Reports tab: Aggregated highlights and diagnostics from the reporting stage.
  • Variant details: Click a variant row to inspect per-tool evidence, scores, and reason fields.

Input Preparation

Input Requirements

  • Accepted files: VCF and VCF.GZ containing human single-nucleotide variants (SNVs). Non-SNV entries are flagged or excluded.
  • Required fields: Standard VCF headers (#CHROM, POS, REF, ALT). Sample columns are accepted if present.
  • Reference build: Must match your inputs and evidence sources to avoid missing IDs.
  • Normalization: Provide left-aligned, normalized SNVs so evidence lookups align with databases.

Run Setup Decisions

  • VCF validation: Fix malformed headers or invalid REF/ALT fields before upload.
  • Evidence failure behavior: Choose Stop or Continue in the Start card before running.
  • Evidence mode: Configured by deployment (online, offline, or hybrid) and reported in the annotation stage stats.

What to Expect

  • Run flow: Upload -> validate -> sequential pipeline stages with live status updates.
  • Run ID: Used for logs, stage status, and result retrieval.
  • Scope: {{ app_name }} supports research and teaching workflows, not clinical diagnosis.

Pipeline Stages

Parser

  • Purpose: Validate the VCF and extract SNV records.
  • Outputs: SNV table plus exclusions for non-SNV or malformed variants.
  • Where to look: Parser tab (counts + exclusions).

Pre-Annotation

  • Purpose: Compute base-change context and nucleotide class labels.
  • Outputs: Base change, transition/transversion, ref/alt class fields.
  • Where to look: Pre-annotation tab.

Classification

  • Purpose: Map VEP consequences into classes (missense, synonymous, nonsense, other).
  • Outputs: Category plus reason code/message.
  • Where to look: Consequence tab and variant details panel.

Prediction

  • Purpose: Run SIFT, PolyPhen-2, and AlphaMissense predictions (when configured).
  • Outputs: Scores, labels, provenance, and reason messages when unavailable.
  • Where to look: Functional prediction tab and variant details panel.

Annotation

  • Purpose: Pull evidence from dbSNP, ClinVar, and gnomAD.
  • Outputs: Evidence records with outcomes, completeness, and timestamps.
  • Where to look: Evidence annotation tab (tables + annotated VCF preview).

Reporting

  • Purpose: Aggregate stage outputs into run-level summaries.
  • Outputs: Highlights, significant results, completeness summaries.
  • Where to look: Reports tab.

Tools and Evidence Sources

Consequence Annotation

  • VEP: Required for classification and prediction. Needs a local cache and plugin paths.
  • SnpEff: Optional local annotation. Missing databases surface as annotation-stage errors.
  • Configuration: Misconfigured caches or paths will fail Classification/Prediction with clear errors.

Functional Prediction

  • SIFT: Substitution impact scores with tolerated/deleterious labels.
  • PolyPhen-2: Probability-based impact predictions with categorical labels.
  • AlphaMissense: Deep learning pathogenicity scores via VEP plugin data.
  • Applicability: Most meaningful for missense variants; others may show not evaluated.

Evidence Databases

  • dbSNP: rsIDs and identity cross-references.
  • ClinVar: Clinical significance assertions and stable identifiers.
  • gnomAD: Population allele frequency context.
  • Source modes: Online or local VCFs. Outcomes distinguish not found vs unavailable.

Evidence Modes and Policy

Modes

  • Online: Query remote APIs for dbSNP, ClinVar, and gnomAD evidence.
  • Offline: Use locally configured, tabix-indexed VCF databases for evidence lookups.
  • Hybrid: Combine local sources with online fallback where configured.
Evidence mode is configured by deployment and reported in the annotation stage stats.

Failure Policy

  • Stop: Fail the annotation stage and block reporting until evidence issues are resolved.
  • Continue: Allow the run to proceed with evidence marked as partial or unavailable.
  • Where to set: Choose this in the Start card before running or before retrying Annotation.
Evidence completeness is tracked per source and summarized in Reporting and the Results tables.

Results Interpretation and Provenance

Where to Look

  • Progress tab: Run summary, stage timeline, logs, retry/cancel actions.
  • Results tab: Stage outputs, evidence diagnostics, and reporting summaries.

How to Read Results

  • Reports Summary: Consolidated highlights and major findings.
  • Variant table: Per-variant classification, prediction, evidence, and completeness.
  • Stage tabs: Trace how each stage produced its outputs.
  • Variant details: Click a row to see per-tool scores, evidence, and reasons.

Completeness and Provenance

  • Completeness: Complete = all sources responded, Partial = some missing, Unavailable = none responded.
  • Outcomes: Each tool/source reports success, not found, unavailable, or error.
  • Reasons: Human-readable messages explain missing data or configuration gaps.
  • Provenance: Records include tool/source labels and retrieval timestamps.
All tables and controls are keyboard-navigable with visible focus styles.

Troubleshooting and FAQ

  • Validation fails immediately: Fix malformed headers or invalid REF/ALT SNVs.
  • No runs visible: Upload a VCF and press Start. The run appears after validation.
  • Classification fails: Verify VEP configuration and cache paths, then retry.
  • Prediction missing outputs: Check VEP plugin setup (SIFT, PolyPhen-2, AlphaMissense).
  • Evidence sources unavailable: Check connectivity or local VCF paths; retry Annotation.
  • gnomAD lookup errors: Verify API connectivity or local dataset compatibility.
  • Partial results: Use Evidence diagnostics to see which sources failed and why.
  • Reference build mismatch: Re-run with the correct build to improve evidence matches.
